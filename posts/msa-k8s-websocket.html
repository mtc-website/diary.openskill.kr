🚇 Ingress 터널 시뮬레이터

현재 단계 대기중
요청 카운트 0

진입 경로 (7단계)

  1. 1. Browser → 주소창 입력
    http://metacoding.local
  2. 2. /etc/hosts 조회 → 127.0.0.1 로 해석
  3. 3. 127.0.0.1:80 접속 → minikube tunnel 이 수신
  4. 4. 터널 통과 → Minikube 컨테이너 내부로 전달
  5. 5. Node 의 80 포트 = Ingress Controller Pod (hostPort)
  6. 6. Ingress Rule 매칭 → ClusterIP Service 선택
  7. 7. Service → Pod (로드밸런싱) → 응답
💡 핵심
• NodePort / LoadBalancer 없이 외부 진입 가능.
• Docker driver는 minikube tunnel 필요.
• VM driver (kvm2/virtualbox)라면 minikube ip 로 직접 접근 OK.
• Ingress Controller도 결국 Pod → Node 내부에 위치.

🔁 멱등성 시나리오

현재 모드 멱등성 ON
큐 대기 메시지 0
💡 messageId 의 역할
• Kafka는 at-least-once 전달 → 네트워크 재시도·리밸런싱 시 같은 메시지가 2번 이상 도착할 수 있음
• 소비자가 qty -= 1 같은 "상태 변경" 로직을 그냥 돌리면 → 재고 중복 차감 BUG
• 해결: 메시지에 messageId(= 사가 단계별 고유 ID) 부여 → Consumer가 처리 직전
  processed_message_tb에서 "이 messageId 처리했나?" 조회 → 있으면 skip
• 비즈니스 UPDATE + processed_message INSERT 를 같은 트랜잭션으로 묶어야 원자적으로 동작

🧪 실험 방법
1. ON 상태에서 "새 메시지" → "동일 메시지 재전송" → ▶ 두 번 소비 → 재고 1번만 차감 ✅
2. OFF 상태로 토글 → 같은 순서 반복 → 재고 2번 차감되는 BUG 재현 ❌
