WebSocket 프로토콜 시뮬레이터

Spring Boot STOMP WebSocket — 5단계 진화 + FCM 비교

1. HTTP → WebSocket Upgrade 핸드셰이크

WebSocket은 독립 프로토콜이 아니라 HTTP 위에서 Upgrade 헤더를 통해 연결을 전환합니다.
클라이언트가 Upgrade: websocket 헤더로 요청 → 서버가 101 Switching Protocols 응답 → TCP를 WebSocket으로 승격합니다.

TCP 연결 HTTP Request 서버 검증 101 Response WS 연결 수립 STOMP CONNECT STOMP CONNECTED
Browser (Client)
new WebSocket('ws://host/connect')
Spring Boot Server
@EnableWebSocketMessageBroker
대기 중 — 버튼을 눌러 핸드셰이크를 시작하세요

2. HTTP POST → Spring Controller → Simple Broker

웹소켓 메시지가 아닌 일반 HTTP POST 요청으로 메시지를 보냅니다.
Spring Controller에서 DB 저장 후 SimpMessageSendingOperations.convertAndSend()로 Simple Broker에 전달합니다.
이것이 현재 프로젝트의 ChatController.save() 메서드가 동작하는 방식입니다.

→
HTTP 요청 (단방향, 비연결)
매 요청마다 새로운 TCP 연결을 맺고 응답 후 끊어집니다.
클라이언트 → 서버 한 방향만 가능. 서버가 먼저 보낼 수 없습니다.
form POST /chat → 서버 내부에서 WebSocket Broker를 통해 구독자에게 Push합니다.
핵심: 클라이언트가 WebSocket이 아닌 HTTP POST로 메시지를 보내고, 서버 내부에서 sms.convertAndSend()로 WebSocket 구독자에게 브로드캐스트합니다.
// ChatController.java — 현재 프로젝트 코드
@PostMapping("/chat")
public String save(String msg) {
    Chat chat = chatService.save(msg); // DB 저장
    sms.convertAndSend("/sub/chat", chat); // ← Broker에 전달!
    return "redirect:/";
}

3. STOMP 직접 → Simple Broker (Controller 스킵)

클라이언트가 STOMP로 /sub 프리픽스 목적지에 직접 메시지를 보내면,
Simple Broker가 prefix 매칭하여 Controller를 거치지 않고 바로 구독자에게 릴레이합니다.
DB 저장, 검증 등 비즈니스 로직 없이 순수하게 메시지만 중계합니다.

⇄
WebSocket 전이중 연결 (Full-Duplex, 지속 연결)
1번 탭에서 수립된 WebSocket 연결이 이미 열려 있는 상태입니다.
같은 TCP 소켓 위에서 클라이언트 ↔ 서버 양방향 동시 통신 가능.
HTTP처럼 매번 연결을 맺지 않고, 이미 열린 터널을 통해 STOMP 프레임을 보냅니다.
핵심: 목적지가 /sub로 시작 → Simple Broker가 직접 처리 → @MessageMapping 안 탐!
비즈니스 로직 불가, 단순 릴레이만 가능합니다.
// JavaScript 클라이언트 코드 (WebSocket 전이중 연결 위에서 실행)
stompClient.send("/sub/chat", {}, "hello");
// → 이미 열려있는 WebSocket 터널을 통해 STOMP 프레임 전송
// → /sub 프리픽스 → Simple Broker가 바로 구독자에게 전달
// → Spring Controller(@MessageMapping) 거치지 않음!

4. /pub → @MessageMapping + @SendTo → Simple Broker

클라이언트가 /pub/room으로 STOMP 메시지를 보내면,
setApplicationDestinationPrefixes("/pub") 설정에 의해 prefix /pub이 제거되고 @MessageMapping("/room")이 매칭됩니다.
@SendTo("/sub")로 리턴값이 Simple Broker에 전달됩니다.

⇄
WebSocket 전이중 연결 (Full-Duplex, 지속 연결)
1번 탭에서 수립된 WebSocket 연결이 이미 열려 있는 상태입니다.
stompClient.send("/pub/room")는 이미 열린 WebSocket 터널을 통해 STOMP 프레임으로 전송됩니다.
HTTP POST와 달리 새로운 연결을 맺지 않습니다.
핵심: /pub/room → prefix 제거 → @MessageMapping("/room") → return 값이 @SendTo("/sub")로 전달
SimpleAnnotationMethodMessageHandler가 @MessageMapping을 찾아 메서드를 호출합니다.
// JavaScript (이미 연결된 WebSocket 위에서 STOMP 프레임 전송)
stompClient.send("/pub/room", {}, "1");
// → WebSocket 전이중 터널을 통해 STOMP SEND 프레임 전송

// Spring Server
@SendTo("/sub")
@MessageMapping("/room")
public String pubTest2(String number) {
    System.out.println("나 요청돼? : " + number);
    return "hello world"; // ← 이 리턴값이 @SendTo("/sub")로 전달!
}

5. /pub → @MessageMapping → convertAndSend("/sub/"+number) ★ 최종

최종 구조! 클라이언트가 /pub/room으로 보내면 Controller에서 sms.convertAndSend("/sub/" + number)로 동적 목적지에 전달합니다.
이 구조로 구독자를 세분화할 수 있습니다. (채팅방 1, 채팅방 2 등)

⇄
WebSocket 전이중 연결 (Full-Duplex, 지속 연결)
1번 탭에서 수립된 WebSocket 연결이 이미 열려 있는 상태입니다.
stompClient.send("/pub/room")는 이미 열린 WebSocket 터널을 통해 STOMP 프레임으로 전송됩니다.
동적으로 "/sub/" + number 목적지를 결정하여 구독자를 세분화할 수 있는 최종 구조입니다.
핵심 (최종 구조): Controller에서 convertAndSend("/sub/" + number)로 동적으로 목적지를 결정합니다.
→ /sub/1 구독자에게만, /sub/2 구독자에게만 선택적 전달 가능!
// JavaScript (이미 연결된 WebSocket 위에서 STOMP 프레임 전송)
stompClient.send("/pub/room", {}, "1");
// → WebSocket 전이중 터널을 통해 STOMP SEND 프레임 전송

// Spring Server — /pub/room 으로 요청이 들어오면
@MessageMapping("/room")
public void pubTest1(String number) {
    System.out.println("나 요청돼? : " + number);
    sms.convertAndSend("/sub/" + number, "hello world " + number);
    // ← number가 "1"이면 /sub/1 구독자에게만 전달!
    // ← number가 "2"이면 /sub/2 구독자에게만 전달!
}

왜 이게 최종 구조인가?
• 4번(@SendTo)은 고정 목적지 → 모든 구독자에게 동일 메시지
• 5번(convertAndSend)은 동적 목적지 → /sub/1, /sub/2 등 구독자 그룹을 분리 가능
• 채팅방, 게임 방, 알림 채널 등 구독자 세분화가 필요한 모든 실시간 서비스에 필수!

6. Firebase Cloud Messaging (FCM) — 모바일 푸시 알림 🔔

앱이 꺼져있어도 휴대폰에 알림을 전달하려면 Firebase Cloud Messaging(FCM)을 사용합니다.
알림을 보내는 쪽은 HTTP POST 단방향, 알림을 받는 휴대폰은 FCM과 상시 연결된 구조입니다.
보내는 사람은 응답이 필요 없고 단발성이므로 전이중 연결이 필요 없습니다.

→
발신: HTTP POST (단방향, 비연결)
Admin/Backend 서버가 POST https://fcm.googleapis.com/v1/.../messages:send로 알림 요청.
단발성 요청이라 상시 연결이 필요 없음 → HTTP가 적합.
⇄
수신: 휴대폰 ↔ FCM 상시 연결 (Persistent)
휴대폰은 부팅 시 FCM 서버와 상시 연결을 유지합니다 (XMPP/HTTP2 기반).
서버가 먼저 Push를 보내야 하므로 양방향/상시 연결이 필수.
핵심 비교: 4·5번 탭의 WebSocket 구조와 비슷하지만, Publisher(발신자)가 HTTP POST로 전환된 구조입니다.
→ 1:N Push를 보낼 때 보내는 사람까지 상시 연결을 유지할 필요는 없다는 것이 포인트!
// ① 발신: Backend 서버 (Java / Node 등) — HTTP POST 한 번
POST https://fcm.googleapis.com/v1/projects/{PID}/messages:send
Authorization: Bearer {ACCESS_TOKEN}

{ "message": {
  "token": "device_registration_token",
  "notification": {
    "title": "새 메시지",
    "body": "안녕하세요!"
  }
} }

왜 Publisher 쪽은 HTTP여도 될까?
• 발신자는 한 번 알림 요청만 하면 끝 → 서버의 응답을 기다릴 필요도 없음
• 수신자(휴대폰)는 언제 알림이 올지 모름 → 서버가 먼저 밀어 넣으려면 상시 연결 필요
• FCM이 디바이스 토큰별 연결 풀을 내부적으로 유지해 대신 Push
WebSocket STOMP vs FCM 비교:
• WebSocket (4·5번 탭): 브라우저가 열려있는 동안만 실시간 통신 가능
• FCM (6번 탭): 앱이 꺼져있어도 OS 수준에서 알림 수신 가능
• 공통점: 수신자는 상시 연결이 필요하다는 것 (연결 주체만 다름)
